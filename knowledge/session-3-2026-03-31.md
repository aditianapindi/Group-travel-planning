# Session 3 — March 31, 2026

## What happened
Stress-tested the core flow and made significant UX improvements. Consolidated duplicate project folders. Shifted from "build more features" to "verify what exists."

## Changes made

### Folder consolidation
- Merged ~/group-travel-planning/ (research-only) into ~/nod/ (code + research)
- Deleted ~/group-travel-planning/ after verifying all 27 files present via MD5 hashes
- Single project directory now: ~/nod/

### Participant motivation design (new)
- Added MotivationBlock above participant form: shows who's responded by name, "takes 30 seconds" nudge, deadline with consequence ("majority decides without you")
- Simplified form: core fields (name, RSVP, destination vote, budget) up front, headcount/group type/kids behind "More details (optional)" toggle
- Better confirmation: organizer sees "Your preferences are saved" + action guidance, participant sees names + "organizer will lock once everyone's in"
- Deadline urgency: status bar now shows "Votes close Thursday" instead of "3d left", highlights when <24h

### Organizer experience
- Name pre-filled from trip creation, RSVP hidden (always yes)
- Shows "You're the organizer — just vote on your preferences below"
- Empty state when 0 responses: "No responses yet. Share the link above."
- Guidance when <2 confirmations: "1 of 2 confirmations needed to lock"

### Share bar
- WhatsApp: now an icon button with brand green (#25D366)
- Added iMessage share button (Apple green #34C759)

### Realtime updates
- Replaced polling with Supabase Realtime subscription
- Filters to INSERT events on participants table for specific trip
- Deduplicates if submitter's own insert triggers callback
- Error handler logs on CHANNEL_ERROR

### Itinerary improvements
- Added TripSummary at bottom: parses costs from selected options, shows estimated total per person
- Shows whether total is within/over group's budget range
- "What's next?" section: organizer told to share + customize, participant told to reach out to organizer

### Quality fixes
- Gemini API: 30-second timeout via AbortController
- Lock button: persistent error message + retry (not alert)
- Root error.tsx: "Something went wrong" + "Try again" button
- Root loading.tsx: spinner
- Background: #FFFFFF → #FAF9F7 (warm off-white)
- Budget label: clarified as "total trip budget" with "Include travel, stay, food, and activities"
- Unified error message for all API failures: "Couldn't generate the plan. Tap to try again."

## Decisions made
- **Holiday calendar**: Designed but NOT built. Documented in knowledge/decision-holiday-calendar.md. Right idea, wrong moment — core flow needed verification first. Describe in PRD.
- **Participant motivation**: Built. Documented in knowledge/decision-participant-motivation.md. Behavioral levers: named accountability > deadline consequence > tiny ask > instant feedback.
- **Trip probability meter**: Rejected. Can't calculate honestly, feels like fake gamification.
- **Destination price estimates**: Deferred. Would need Gemini API call at trip creation + schema change. Describe in PRD.
- **Dark mode**: Skipped for deadline. Not expected for prototype submission.
- **Polling vs Realtime**: Switched from polling to Supabase Realtime. Polling was the wrong tool — wasteful at scale, Supabase Realtime is built into the SDK.

## Lessons learned
- Verify rendered output for each persona before handoff (caught "Aditican" space bug, wrong tone for organizer, duplicate names)
- "Do less. Verify more" — almost built holiday calendar (3-4 hrs) when core flow wasn't stress-tested
- Push back on own recommendations, not just user's requests
